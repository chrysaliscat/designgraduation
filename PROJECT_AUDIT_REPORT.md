# 📝 农产品平台 - 源码深度审计与优化报告

经过对本系统后端核心业务（特别是 `AgOrderServiceImpl` 等涉及到核心利益的模块）的全面通读和审计，结合您**毕业设计**的背景要求，我为您整理了以下专业评估报告。

---

## 🐞 1. 业务逻辑 Bug (建议立即修复)

### 1.1 空订单漏洞 (“幽灵订单”)
*   **出处**：`AgOrderServiceImpl.java` -> `createOrder()` 方法 (第 78 行)。
*   **现象**：系统仅判断了 `if (StringUtils.isNotEmpty(agOrder.getOrderItemList()))`，但如果前端传入的购物车列表为**空**，代码将跳过库存检查和金额计算分支，直接向数据库插入一条 `TotalAmount = 0` 的空订单记录。
*   **后果**：会产生大量垃圾脏数据，导致商品销量统计报错以及财务计算抛出异常。

### 1.2 粗暴的异常拦截机制
*   **出处**：`AgOrderServiceImpl` (第 96、149 行等)。
*   **现象**：当库存不足或订单状态不允许取消时，直接使用了 Java 底层的 `throw new RuntimeException("库存不足")`。
*   **后果**：该异常会越过若依的全局响应封装，直接给 Vue 前端抛出一个 HTTP 500 的服务器内部报错页面，而不是右上角的红色友好提示框（如 `Message.error`）。

## 🛡️ 2. 系统漏洞隐患 (答辩高频考点)

### 2.1 缺乏“防重放”机制 (高并发下单漏洞)
*   **现象**：在下订单的 `POST /agri/order` 接口上，并没有引入防止表单重复提交的控制。
*   **风险**：在弱网环境下，或者用户疯狂连点“提交订单”按钮，由于网络延迟，即使 MySQL 的 `decreaseStock` 能防止超卖，也**会给该用户生成多个重复的订单记录**。
*   **解决方案**：需在接口上使用若依的 `@RepeatSubmit` 注解，基于 Redis 加锁防止短时间内重复下单。

### 2.2 数据隔离策略硬编码
*   **出处**：`AgOrderController` -> `list()` (第 53 行)。
*   **现象**：为了区分商家和消费者看到的订单，代码中使用了硬编码校验：`"agri_buyer".equals(r.getRoleKey())`。
*   **风险**：一旦您在后台系统的界面上修改了“买家”角色的标志符，订单过滤系统将完全瘫痪，所有人都能看到所有数据（数据越权）。应该采用更为灵活的 `@DataScope` 数据权限注解机制。

---

## 🚀 3. 急需补充的核心亮点 (毕设加分项)

对于一个以“销售平台”为核心的毕业设计项目，如果仅有目前的增删改查（CRUD），在技术答辩时很难脱颖而出，甚至略显单薄。建议补充以下功能模块以**极大提升项目技术含量**：

### 🌟 亮点 A: 真实支付模拟闭环 (Alipay Sandbox)
*   **现状**：目前点击“购买”，订单直接变为“已完成”或“待发货”。
*   **建议补充**：强烈建议引入 **“支付宝沙箱支付环境”**。当用户点击付款时，弹出一个网页二维码，由手机扫码进行模拟付款（沙箱不需要真实扣钱，是开发者用的假余额）。
*   **加分理由**：评委老师非常看重电商项目是否具有业务闭环，这能体现你掌握了第三方 API 的对接能力。

### 🌟 亮点 B: 引入 Redis 处理“高并发秒杀库存”
*   **现状**：目前系统的扣库存是直接打向 MySQL 库的（`update ag_product set stock = stock - 1`）。
*   **建议补充**：你可以设计一个“农产品限时助农秒杀”活动。告诉评委老师：“为了防止高并发下 MySQL 宕机，我首先在 Redis 中预扣减了库存队列，然后再异步写入 MySQL”。
*   **加分理由**：这不仅填补了 Redis 当前在业务中使用较少的空白，还直接点亮了“高并发架构”的技术树，是拿高分的绝对利器。

### 🌟 亮点 C: 定时任务自动取消未支付订单
*   **现状**：如果没有支付模块，订单永远挂在那里。如果加了支付，就需要处理“30分钟不付款自动取消”。
*   **建议补充**：利用框架内置的 `Quartz` 模块（项目里已经有 `ruoyi-quartz` 了，但没怎么用），写一个定时轮询任务，自动释放超时未支付订单的库存。
